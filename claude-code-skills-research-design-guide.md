# Claude Code Skills — 비코딩 리서치 스킬 설계 가이드
  
[출처](https://zenn.dev/s4kura/articles/claude-code-indie-idea-scout-skill )   
   
---

## **개요:**
Claude Code의 Skills는 프롬프트를 패키지화해 재사용하는 구조다. 프론트엔드 디자인 생성이나 코드 리뷰에 주로 사용되지만, **코드를 작성하지 않는 리서치·분석 작업에도 응용할 수 있다.** 이 글에서는 "개인 개발자를 위해 최신 기술 트렌드에서 틈새 비즈니스 아이디어를 발굴하는" 스킬(`indie-idea-scout`)을 직접 설계·실행한 결과를 정리한다.

---

## **스킬 전체 구조:**
파일은 `~/.claude/skills/` 아래에 다음과 같이 배치한다.

```
indie-idea-scout/
├── SKILL.md                        # 스킬 정의 (4페이즈 실행 절차)
└── references/
    ├── evaluation-framework.md     # 5축 평가 루브릭 + 제외/경고 필터
    ├── market-analysis-patterns.md # 개인 개발 성공 패턴 6종 정의
    └── search-strategy.md          # WebSearch 쿼리 템플릿 모음
```

`/indie-idea-scout`를 입력하는 것만으로 다음 4페이즈가 자동 실행된다.

```
Phase 1: 트렌드 리서치 (WebSearch 10~14회)
    ↓
Phase 2: 갭 분석 (3종류의 갭 특정)
    ↓
Phase 3: 아이디어 생성 + 5축 평가
    ↓
Phase 4: 구조화 리포트 출력
```

---

## **설계 핵심 포인트:**

**SKILL.md는 "실행 절차"에 집중한다**

SKILL.md의 frontmatter는 단순하게 유지하고, 본문에는 4개 페이즈의 실행 순서와 출력 포맷만 기술한다. 핵심은 **"무엇을"이 아니라 "어떻게 조사하는가"를 지시하는 것**이다.

예를 들어 Phase 1 트렌드 리서치에서는 다음 4개 조사 카테고리를 명시한다.

- **신규 API·플랫폼 변경** — 최근 공개된 API
- **소비자 불만** — Reddit 등의 페인포인트
- **성장 중인 틈새 시장** — Indie Hacker 사례
- **규제·사회 변화** — 법 개정이 만들어 내는 신규 수요

처음에는 카테고리 없이 "최신 트렌드를 조사해줘"라고만 작성했다. 그 결과 Claude는 AI 관련 뉴스를 3~4건 수집하고 종료했다. **카테고리를 명시하자 검색이 10회 이상으로 늘어났고, 소비자 불만이나 규제 변화 같은 "기술 뉴스 이외의 각도"에서도 갭을 발견하게 되었다.**

**references/에 "판단 기준"을 분리한다**

| 파일 | 역할 |
|------|------|
| `evaluation-framework.md` | 5축 스코어링 채점 기준 + 제외/경고 필터 |
| `market-analysis-patterns.md` | 개인 개발에서 성공하기 쉬운 6가지 패턴 정의 |
| `search-strategy.md` | WebSearch 쿼리 템플릿 모음 |

SKILL.md 한 곳에 전부 쓰지 않는 이유가 있다. **프롬프트가 비대해지면 Claude는 전체를 균일하게 참조하지 않고, 후반부 지시를 가볍게 여기는 경향이 있다.** 상세 기준은 별도 파일로 분리하고, SKILL.md에서는 `references/evaluation-framework.md를 참조할 것`이라고 한 줄로 지시하는 편이 더 확실하게 읽힌다.

## **개선 전후 비교:**

```
- 첫 번째 버전 (references/ 없음):
  "AI를 활용한 이미지 편집 툴이 흥미롭습니다"
  "챗봇 커스터마이즈 수요가 있습니다"
  → 추상적. 누구나 생각할 수 있는 수준. 스코어도 없어 우선순위 불명.

+ 개선 후 (references/ 3파일 추가):
  "Seedance 2.0 API가 2/10 공개. EC셀러 특화 동영상 생성은 미등장"
  "평가: 타이밍 5, 실현가능성 5, 종합 4.4/5.0"
  → 구체적. 근거 URL 포함. 스코어로 우선순위 명확.
```

품질 향상에 가장 크게 기여한 두 가지는 다음과 같다.

**평가 프레임워크의 효과**: 5축 평가를 넣기 전에는 "흥미로운 아이디어"가 출력되었다. 넣은 후에는 "흥미로우면서 개인 개발자가 지금 당장 착수해 수익화할 수 있는 아이디어"로 바뀌었다. 특히 **"개인 개발 적합성" 축**이 핵심으로, 이 축이 없으면 팀이 필요한 아이디어나 운용 부담이 큰 아이디어도 섞여 들어온다.

**검색 전략 가이드의 효과**: 쿼리 템플릿이 없으면 Claude는 영어로 1~2회 검색하고 끝낸다. 템플릿을 제공하자 "영어로도 일본어로도 검색하자", "Reddit에서 페인포인트도 조사하자"는 식으로 동작이 변하고, 검색 횟수가 3~4회에서 10회 이상으로 늘었다. **정보량의 차이가 그대로 아이디어 품질로 이어진다.**

## **실행 결과 — 2026년 2월 기준 출력된 3개의 프로덕트 제안:**
WebSearch로 발견된 주요 트렌드는 다음과 같다. Seedance 2.0 API(ByteDance, 2월 10일 릴리즈, $0.10~0.80/분), Gemini 3 Computer Use API(구글 브라우저 자동 조작 프리뷰), 미국 프라이버시 규제 강화(콜로라도 AI법 2026년 6월 시행 예정), AI 일관 캐릭터 생성 기술의 실용화가 감지되었다.

**아이디어 1: ClipForge — 상품 쇼트 동영상 생성기 (종합 4.4/5.0)**

상품 사진을 업로드하는 것만으로 TikTok / Instagram Reels용 쇼트 동영상을 자동 생성하는 웹앱이다.

| 평가 축 | 스코어 |
|---------|--------|
| 타이밍 | 5 — Seedance API 공개 직후. EC 특화 동영상 툴은 미등장 |
| 실현가능성 | 5 — OpenAI 호환 API 래퍼 개발으로 구현 용이 |
| 수익성 | 4 — 월 ¥980~1,980. 100명으로 월 12만 엔 |
| 경쟁 우위 | 4 — "EC셀러 특화"는 미개척 |
| 개인 개발 적합성 | 4 — API 래퍼형으로 지원 비용 낮음 |

**패턴 판정**: API 래퍼형. 타임윈도우는 1~3개월(API 공개 직후가 승부처).

**Why Now**: Seedance 2.0이 2월 10일 릴리즈되어 이미지→동영상 변환이 $0.10/분부터 사용 가능. 개인 셀러는 "동영상이 없으면 팔리지 않는다"고 느끼지만 편집 스킬이 없다.

**아이디어 2: KomaStudio — 만화/Webtoon AI 캐릭터 엔진 (종합 4.0/5.0)**

AI로 만화 캐릭터를 일관된 화풍으로 여러 컷에 걸쳐 생성할 수 있는 툴이다. 캐릭터 시트를 한 번 만들면 다양한 포즈·표정을 즉시 생성할 수 있다.

| 평가 축 | 스코어 |
|---------|--------|
| 타이밍 | 4 — 캐릭터 일관성 기술이 성숙기 |
| 실현가능성 | 3 — LoRA/IP-Adapter 학습 비용 존재 |
| 수익성 | 5 — 월 ¥1,480~2,980. 100명으로 월 18만 엔 |
| 경쟁 우위 | 5 — 만화/Webtoon 특화 직접 경쟁자 없음 |
| 개인 개발 적합성 | 3 — 이미지 생성 인프라 관리 필요 |

**패턴 판정**: 커뮤니티 특화형. 타임윈도우는 6~12개월로 길다.

**Why Now**: Neolemon, LTX Studio 등 캐릭터 일관성 툴이 등장했으나 모두 서양 화풍(픽사 스타일) 중심. 일본 만화·Webtoon 화풍에 특화된 툴은 공백 지대다.

**아이디어 3: PriceHound — AI 탑재 웹 변경 모니터링 (종합 3.6/5.0)**

웹사이트 변경을 감시하고, AI가 "가격이 ¥2,980→¥1,980으로 인하됐습니다"처럼 요약해 LINE으로 알림을 보내는 툴이다.

| 평가 축 | 스코어 |
|---------|--------|

| 실현가능성 | 4 — 기술적으로 기존 패턴 |

**패턴 판정**: 가격 파괴형. Visualping($14/월)의 1/3 가격으로 일본어 특화.

## **자신만의 리서치 스킬을 만드는 3가지 원칙:**

**원칙 1: SKILL.md에는 판단 기준을 쓰지 않는다**

SKILL.md에 쓰는 것은 **워크플로우(무엇을 어떤 순서로 할 것인가)** 만이다. "좋은 아이디어란 무엇인가", "신뢰할 수 있는 소스란 무엇인가" 같은 판단 기준은 `references/`에 분리한다. SKILL.md가 길어지면 Claude가 후반부 지시를 경시하고, 판단 기준만 바꾸면 다른 용도로 전용할 수 있기 때문이다. 예를 들어 `evaluation-framework.md`를 "기술 아티클 품질 기준"으로 교체하면 동일한 4페이즈 구조로 "기술 아티클 아이디어 발굴 스킬"을 만들 수 있다.

**원칙 2: 모호한 검색을 시키고 싶지 않다면 쿼리의 "형식"을 제공한다**

WebSearch를 사용하는 스킬에서 가장 차이가 나는 부분이다. Claude에게 "최신 트렌드를 조사해줘"라고 하면 1~2회 검색으로 만족한다. **카테고리별 쿼리 템플릿을 제공하면** 검색 횟수와 정보의 다각성이 극적으로 달라진다.

템플릿을 만들 때의 핵심 포인트는 다음과 같다. 영어와 한국어(일본어) 두 가지를 모두 준비하고, `site:reddit.com`이나 `site:news.ycombinator.com`처럼 소스를 지정해 생생한 목소리를 수집한다. 연도를 반드시 포함해(`{year}` 플레이스홀더 활용) 최신 정보를 취득하게 한다.

**원칙 3: 스코어링 기준을 넣으면 "흥미롭다"가 "실행 가능하다"로 바뀐다**

평가 축이 없으면 Claude는 "흥미롭지만 실현 곤란"하거나 "흥미롭지만 경쟁이 포화"된 아이디어도 출력한다. 스코어링 기준을 넣으면 **각 아이디어에 점수와 근거가 붙는다.** 근거가 약한 아이디어는 자연스럽게 낮은 점수를 받아 필터링된다. 평가 축은 자신의 상황에 맞게 설계하는 것이 중요하다. 예를 들어 "부업으로 하고 싶다"면 "주말만으로 운용 가능한가" 축을 추가하는 식의 조정이 유효하다.

## **응용 가능한 스킬 예시:**

동일한 설계 패턴으로 만들 수 있는 스킬이다.

| 스킬명 | Phase 1 | references/ |
|--------|---------|------------|
| 기술 아티클 아이디어 발굴 | Zenn/Qiita 트렌드 + HackerNews 검색 | "버즈 아티클"의 평가 기준 |
| 경쟁 분석 | 특정 장르의 기존 서비스 망라 조사 | SWOT 분석 템플릿 |
| 기술 선정 리서치 | 프레임워크/라이브러리 최신 동향 조사 | 선정 기준(성능, 학습 비용 등) |
| 발표 아이디어 발굴 | 컨퍼런스 CFP + 최근 화제 | "통과하는 프로포절"의 평가 기준 |

공통 구조는 **"WebSearch로 정보 수집 → 독자적 평가 기준으로 필터링 → 구조화 리포트 출력"** 이 3단계다. SKILL.md에서 수집 카테고리를 정하고, `references/`에 평가 기준을 배치한다. 이 구조만 갖추면 코드를 한 줄도 쓰지 않고 Claude의 분석 능력을 확장할 수 있다.

## **총정리:**

Claude Code의 Skills는 코드 생성뿐 아니라 리서치와 분석에도 충분히 사용 가능하다. 핵심 포인트를 정리하면 다음과 같다.

- **SKILL.md는 워크플로우, references/는 판단 기준** — 이 분리가 스킬의 재현성과 품질을 결정한다
- **Claude의 검색은 "형식"으로 제어한다** — 카테고리 분류 + 쿼리 템플릿으로 정보의 망라성이 극적으로 달라진다
- **스코어링 기준이 출력 품질을 좌우한다** — "흥미롭다"가 아니라 "실행 가능하고 수익화할 수 있다"로 필터링된다
- **처음에는 대충 만들어도 된다. references/를 추가할수록 품질이 올라간다** — 한 번에 완성시키려 하지 말고 실행 → 개선 사이클을 돌린다

스킬 파일 구성은 `SKILL.md` + `references/` 3파일, 합계 약 20KB다. 한 번 만들면 몇 번이든 재실행할 수 있으며, 매번 최신 WebSearch 결과에 기반한 제안을 얻을 수 있다. **"이런 리서치를 매번 하고 있다"고 느낀다면, 그것이 스킬화의 후보다.**
