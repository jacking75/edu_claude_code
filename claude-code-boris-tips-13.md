# Claude Code 개발자 Boris의 활용 팁 13가지

[출처](https://zenn.dev/explaza/articles/a387d2bf1cb448 )  
  
> 이 내용은 Claude Code의 창시자 Boris가 직접 공개한 자신과 팀의 실제 활용법이다.


## **1. 터미널에서 Claude 5개 병렬 실행**
탭에 1~5번 번호를 붙여 관리한다. Claude가 입력을 기다릴 때 시스템 알림으로 통보받도록 설정해 효율을 높인다.


## **2. 로컬과 병행해서 Web 버전도 활용**
[claude.ai/code](https://claude.ai/code) 에서 5~10개의 Claude를 동시에 실행한다. 로컬 터미널에서 코딩하면서 `&`를 사용해 로컬 세션을 Web 버전으로 이어받거나, `--teleport`로 오가기도 한다. 매일 아침과 하루 동안 iPhone의 Claude 앱에서 세션을 시작하고 나중에 확인하는 방식으로도 사용한다.


## **3. Opus 4.5 (thinking 포함)를 모든 작업에 사용**
지금까지 사용해본 코딩 모델 중 최고라고 평가한다. Sonnet보다 크고 느리지만, **지시 수정 횟수가 줄고 툴 사용도 우수**하기 때문에 결과적으로 작은 모델보다 거의 항상 더 빠르게 마무리된다.


## **4. 팀 전체가 하나의 CLAUDE.md를 공유**
Claude Code 리포지토리에 `git`으로 체크인해서 관리한다. 팀원 전원이 주 수 차례씩 기여하며, **Claude가 실수를 할 때마다 CLAUDE.md에 추가해 같은 실수를 반복하지 않도록** 한다. 다른 팀은 각자 독립적인 CLAUDE.md를 관리하고, 각 팀이 최신 상태를 유지하는 책임을 진다.


## **5. 코드 리뷰 중 @.claude 태그 활용**
동료의 PR에 `@.claude`를 태그해 PR의 일부로서 `CLAUDE.md`에 내용을 추가한다. **Claude Code GitHub Action** (`/install-github-action`)을 사용하며, 이는 @danshipper의 "Compounding Engineering" 개념을 팀 방식으로 적용한 것이다.


## **6. 대부분의 세션을 Plan 모드로 시작**
`Shift+Tab`을 2번 눌러 Plan 모드에 진입한다. Pull Request 작성이 목표일 때 Plan 모드를 사용하며, **마음에 드는 계획이 나올 때까지 Claude와 주고받은 후**, 편집 자동 승인 모드로 전환하면 Claude가 보통 1회에 완료한다. 좋은 계획은 결과 품질에 결정적인 영향을 준다.


## **7. 반복 업무 흐름에는 슬래시 커맨드 사용**
하루에도 수차례 반복하는 사내 업무 흐름에 적용한다. 반복적인 프롬프트 입력이 불필요해지고, Claude도 이 워크플로우를 활용할 수 있게 된다. 커맨드는 `git`에 체크인하고 `.claude/commands/`에 저장한다.

- **예시**: `/commit-push-pr` 슬래시 커맨드를 매일 수십 회 사용
- 인라인 bash로 `git status` 등의 정보를 사전에 계산해 모델과의 불필요한 왕복을 줄인다


## **8. 서브에이전트를 정기적으로 사용**
슬래시 커맨드와 마찬가지로, 대부분의 PR에서 공통적으로 수행하는 워크플로우를 자동화하는 도구로 활용한다.

- **`code-simplifier`**: Claude 작업 후 코드를 단순화
- **`verify-app`**: Claude Code를 엔드투엔드로 테스트하는 상세 절차


## **9. PostToolUse 훅으로 코드를 자동 포맷**
Claude는 대부분 잘 정형화된 코드를 생성하지만, **훅이 나머지 10%를 처리**한다. 이를 통해 나중에 CI에서 포맷 에러가 발생하는 것을 사전에 방지한다.


## **10. `--dangerously-skip-permissions`는 사용하지 않는다**
대신 `/permissions`를 사용한다. 환경에서 안전하다고 판단되는 일반적인 bash 커맨드를 사전에 허가해 불필요한 권한 프롬프트를 피한다. 대부분의 설정은 `.claude/settings.json`에 체크인해 팀과 공유한다.


## **11. Claude Code가 모든 툴을 사용하게 한다**
MCP 서버와 CLI 툴을 통해 Claude가 외부 서비스에 직접 접근하도록 구성한다. 관련 설정은 `.mcp.json`에 체크인해 팀과 공유한다.

- **Slack** (MCP 서버 경유): 검색 및 메시지 전송
- **BigQuery**: `bq CLI`를 사용해 쿼리를 실행하고 분석 질문에 답변
- **Sentry**: 에러 로그 취득


## **12. 장시간 실행 태스크에 대한 대응**
장시간 실행 작업에서 Claude가 사용자에 의해 블로킹되지 않고 작업을 이어갈 수 있도록 아래 방법을 사용한다.

- **(a)** 완료 시 백그라운드 에이전트로 작업을 검증하도록 Claude에 프롬프트 전송
- **(b)** 에이전트의 **Stop 훅**을 사용해 보다 확정적으로 검증
- **(c)** **ralph-wigum 플러그인** (@GeoffreyHuntley 고안) 사용
- 샌드박스 내에서 `--permission-mode=dontAsk` 또는 `--dangerously-skip-permissions`를 사용해 권한 프롬프트를 회피


## **13. 가장 중요한 팁 — Claude에게 작업을 검증하는 방법을 제공하라**
Boris가 꼽는 **단 하나의 가장 중요한 팁**이다.

**Claude가 피드백 루프를 가지고 있으면 최종 결과 품질이 2~3배 향상된다.**

Claude는 Chrome 확장을 사용해 [claude.ai/code](https://claude.ai/code)에 추가하는 모든 변경사항을 직접 테스트한다. 브라우저를 열고, UI를 테스트하며, 코드가 동작하고 UX가 만족스러울 때까지 반복한다.

검증 방식은 도메인에 따라 달라진다:
- **단순한 것**: bash 커맨드를 실행하는 것만으로 충분
- **테스트 스위트 실행**: 자동화된 테스트로 검증
- **앱 테스트**: 브라우저 또는 폰 시뮬레이터로 실제 동작 확인

이 검증 구조를 견고하게 만드는 데 투자하는 것이 핵심이다.

## **핵심 요약**

| 카테고리 | 주요 팁 |
|---|---|
| 병렬 실행 | 터미널 5개 + Web 5~10개 동시 운용 |
| 모델 선택 | Opus 4.5 (thinking 포함) 전면 사용 |
| 지식 축적 | 팀 공유 CLAUDE.md에 실수마다 추가 |
| 워크플로우 | Plan 모드 → 자동 승인 순서 준수 |
| 자동화 | 슬래시 커맨드 + 서브에이전트 + 훅 |
| 외부 연동 | MCP로 Slack, BigQuery, Sentry 연결 |
| **품질 핵심** | **Claude에게 검증 피드백 루프를 반드시 제공** |
